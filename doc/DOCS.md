## 一、项目简介

TurtleMarket（龟市）是 Turtle WoW 硬核模式下的 **P2P 去中心化交易公告板**。

硬核模式没有拍卖行，玩家只能面对面交易。龟市通过游戏内隐藏聊天频道实现商品发布和求购广播，帮助买卖双方快速匹配，再通过密语联系、面对面完成交易。

| 属性 | 值 |
|------|-----|
| 版本 | 1.1.0 |
| 运行时 | Lua 5.1 · WoW 1.12.1 (Interface 11200) · Turtle WoW |
| 规模 | 10 个 Lua 模块，~4,590 行，纯 Lua 实现（无 XML） |
| 持久化 | `TM_Data`（市场数据）、`TM_PlayerCache`（玩家职业缓存） |
| 命名空间 | `TM`（全局表） |
| 命令 | `/tm` 或 `/market` 打开主面板 |

## 二、功能全景

### 2.1 商品发布

**文件：** `market-post.lua` (305 行)

- **背包选物**：12×6 网格展示背包物品，点击选中，自动读取物品 ID、名称、图标、可用数量
- **Shift+链接**：Hook 了 `ContainerFrameItemButton_OnClick`，支持从背包 Shift+点击填入
- **价格输入**：金/银/铜三个独立输入框
- **备注**：最多 100 字符，支持说明在线时间、小号名等
- **发布流程**：验证 → 生成唯一 ID → 写入本地存储 → 编码 #P 消息 → 入队广播

### 2.2 求购系统

**文件：** `market-browse.lua` 求购 Tab

- **三种选物方式**：
  1. 点击背包物品 — 自动填入名称和图标
  2. 点击聊天链接 — 自动识别物品
  3. 手动输入 — 直接输入物品名称
- **预算上限**：金/银/铜，表示愿意支付的最高价
- **发布**：编码 #W 消息广播到频道

### 2.3 统一浏览

**文件：** `market-browse.lua` (1,123 行) — 最大的模块

主界面 800×700，5 个标签页：浏览 | 出售 | 求购 | 我的商品 | 历史

**浏览页功能：**

| 功能 | 实现 |
|------|------|
| 混合展示 | 出售（绿色 [出售]）和求购（橙色 [求购]）统一列表 |
| 搜索 | 关键词模糊匹配，大小写不敏感 |
| 类型筛选 | 全部 / 出售 / 求购 三个按钮 |
| 价格筛选 | 最低价 ~ 最高价范围 |
| 玩家筛选 | 按卖家/买家名过滤 |
| 只看在线 | 勾选后仅显示 600 秒内有心跳的玩家 |
| 排序 | 价格升序/降序、时间（最新优先）、数量 |
| 分页 | 每页 10 条，底部页码导航 |
| Tooltip | 鼠标悬停显示物品完整属性（SetHyperlink） |
| 密语 | 选中记录后一键发送格式化密语，出售→"密语卖家"，求购→"密语买家" |

**表格列：** 图标 | 类型 | 名称 | 数量 | 价格 | 玩家 | 状态 | 备注

### 2.4 我的商品管理

**文件：** `market-mylistings.lua` (413 行)

- **出售管理**（上半部分，8 行）：显示图标、名称、数量、价格、剩余时间、备注
- **求购管理**（下半部分，6 行）：类似布局
- **取消**：确认弹窗 → 广播 #C/#X 取消消息 → 从本地存储删除
- **重新发布**：复制原价格/数量/备注，生成新 ID 和有效期

### 2.5 交易历史

**文件：** `market-storage.lua`

- 环形缓冲区，最多保留 100 条记录
- 记录内容：操作类型（sold/bought/cancelled）、玩家名、价格、时间
- 在浏览页的"历史"子 Tab 中展示

### 2.6 交易辅助

**文件：** `market-trade.lua` (125 行)

| 功能 | 说明 |
|------|------|
| 价格提示 | 打开交易窗口时自动显示预期价格 |
| 防诈骗警告 | 对方给的金额不匹配时红色警告文字 |
| 自动下架 | 交易完成后自动删除对应挂单 |
| 历史记录 | 交易完成写入 history |
| 密语监控 | 检测 "[龟市]"/"[TurtleMarket]" 格式密语 |
| 音效提示 | 收到交易密语时播放提示音（可关闭） |

### 2.7 信誉系统

**文件：** `libs.lua` (评分计算) + `market-storage.lua` (CRUD) + `market-trade.lua` (评价弹窗)

- **四级评分**：
  - 新手（灰色）→ 可靠（绿色）→ 信赖（蓝色）→ 元老（金色）
- **评价弹窗**：交易完成后弹出好评/中评/差评选项，10 秒倒计时后默认好评
- **存储**：`TM_Data.reputation[playerName]` 记录 positive/negative/neutral 计数

### 2.8 配置面板

**文件：** `market-config.lua` (230 行)

| 选项 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| 过期时间 | 下拉框 | 48 小时 | 可选 24/48/72/168 小时 |
| 缓存上限 | 滑块 | 500 | 范围 100-2000 |
| 心跳间隔 | 下拉框 | 300 秒 | 在线状态广播频率 |
| 密语格式 | 下拉框 | 中文 | 中文 / English |
| 音效开关 | 复选框 | 开启 | 密语提示音效 |
| 启用开关 | 复选框 | 开启 | 整体启用/禁用 |

## 三、技术架构

### 3.1 五层架构

```
┌─────────────────────────────────────────────────────────┐
│  UI 层          browse · post · mylistings · config · trade  │
├─────────────────────────────────────────────────────────┤
│  同步层         market-sync.lua — 心跳 + Gossip 增量同步      │
├─────────────────────────────────────────────────────────┤
│  数据层         market-storage.lua — CRUD · 搜索 · 清理 · 信誉 │
├─────────────────────────────────────────────────────────┤
│  协议层         market-protocol.lua — 编解码 · 分片 · 节流队列  │
├─────────────────────────────────────────────────────────┤
│  核心层         market-core.lua — 初始化 · 频道 · 事件 · 工具函数 │
├─────────────────────────────────────────────────────────┤
│  基础库         libs.lua — 定时器 · 钩子 · UI 组件 · 字符串工具   │
└─────────────────────────────────────────────────────────┘
```

数据流向：**UI → 数据层 → 协议层 → 核心层（频道发送）→ 网络 → 核心层（频道接收）→ 协议层 → 数据层 → UI**

### 3.2 P2P 通信机制

**频道名：** `TurtleMarket`（硬编码）

**频道管理：**
- 登录后自动加入，最多重试 3 次（指数退避 2s/4s/6s）
- 失败后 5 秒再次尝试并警告玩家

**消息隐藏（三层过滤）：**
1. Hook `ChatFrame_OnEvent` — 拦截 CHAT_MSG_CHANNEL 中的 TurtleMarket 消息
2. Hook `ChatFrame_MessageEventHandler` — 二次拦截
3. Hook `DEFAULT_CHAT_FRAME:AddMessage` — 最终防线，模式匹配 `'urtleMarket'` 和 `'#[PCHSWXDF]$'`

**防离开：** Hook `/leave` 命令，阻止玩家手动离开 TurtleMarket 频道

### 3.3 消息协议

**格式：** `#TYPE$payload`

| 类型 | 含义 | 方向 | 载荷格式 |
|------|------|------|----------|
| `#P` | 出售发布 | 广播 | `ID:itemString:name:count:g:s:c:hours:seller:ts:texture:note` |
| `#C` | 取消出售 | 广播 | `ID:sellerName` |
| `#W` | 求购发布 | 广播 | `id:itemId:name:count:maxG:maxS:maxC:buyer:ts:note` |
| `#X` | 取消求购 | 广播 | `ID:buyerName` |
| `#H` | 心跳 | 广播 | `seller:listingCount:ts:vPROTOCOL_VERSION` |
| `#S` | 同步请求 | 广播 | `digest`（格式 `count:hash`） |
| `#D` | 同步数据 | 广播 | `entry1;entry2;...`（每批最多 10 条，`;` 分隔） |
| `#F` | 分片 | 广播 | `msgId$partNum$totalParts$partData` |

**字段转义规则：**

| 原字符 | 转义为 | 用途 |
|--------|--------|------|
| `~` | `~t` | 转义符自身 |
| `:` | `~c` | 字段分隔符 |
| `;` | `~s` | 条目分隔符 |

### 3.4 节流与分片

**发送队列：**
- 优先级排序：取消(0) > 发布(1) > 心跳(2) > 同步(3)
- 队列上限 1,000 条，满时丢弃低优先级
- 每秒处理一条（`ProcessQueue` 由 `TM.timers.every(1, ...)` 驱动）

**节流参数：**

| 参数 | 值 | 说明 |
|------|-----|------|
| `THROTTLE_INTERVAL` | 1 秒 | 最小发送间隔 |
| `BURST_LIMIT` | 3 条 | 连续发送上限 |
| `COOLDOWN_TIME` | 5 秒 | 突发后冷却时间 |

**分片传输：**
- 触发条件：消息 > 250 字符（`MAX_MSG_LEN`）
- 每片有效载荷 ~220 字符（`MAX_MSG_LEN - 30`）
- 格式：`#F$msgId$partNum$totalParts$data`
- 接收端重组：`TM:HandleFragment()` 缓存各片，全部到达后拼接
- 保护：单消息最多 100 片，最多 50 个并发 msgId，60 秒过期清理

**消息清洗（发送前）：**
- `\` → `/`（路径分隔符统一）
- `|` → 删除（防止 WoW 颜色代码注入）

### 3.5 Gossip 同步协议

**心跳广播：**
- 默认间隔 300 秒（可配置）
- 内容：玩家名 + 挂单数 + 时间戳 + 协议版本
- 用途：维持在线状态、让其他节点知道自己有多少商品

**同步流程：**

```
Player A 上线
  ├─ T+3s   发送首次心跳 #H
  ├─ T+10s  发送同步请求 #S（含本地摘要 count:hash）
  ├─ T+15s  广播自己的所有出售 #P
  └─ T+18s  广播自己的所有求购 #W

其他玩家收到 #S
  ├─ 跳过自己的消息
  ├─ 检查冷却（5 分钟内只响应一次）
  ├─ 40% 概率决定是否响应（防风暴）
  ├─ 随机延迟 3-20 秒
  └─ 发送 #D 批量同步数据（每批 10 条）
```

**摘要比对：** `TM:ComputeDigest()` 使用 DJB2 风格哈希对排序后的 listing ID 计算哈希值，格式 `count:hash`。接收方可据此判断数据是否一致。

### 3.6 数据持久化

**SavedVariables 结构：**

```lua
TM_Data = {
    listings = {},       -- 全网出售订单 {id → listing}
    wants = {},          -- 全网求购请求 {id → want}
    myListings = {},     -- 我的出售 {id → listing}
    myWants = {},        -- 我的求购 {id → want}
    history = {},        -- 交易历史（环形缓冲区，最多 100 条）
    reputation = {},     -- 信誉数据 {playerName → {positive, negative, neutral}}
    syncMeta = {},       -- 同步元数据
    config = {           -- 用户配置（持久化）
        enabled = true,
        defaultExpireHours = 48,
        maxListings = 500,
        heartbeatInterval = 300,
        whisperFormat = "cn",
        soundAlert = true,
        browseSort = "time",
        wantSort = "time",
        minimapAngle = 225,
    }
}

TM_PlayerCache = {
    players = {}         -- {playerName → classString}，上限 1000
}
```

**过期清理（`TM:CleanupListings()`）：**
- 触发时机：登录时 + 每 30 分钟
- 清理条件：
  1. 超过 `expiresAt` 时间戳
  2. 72 小时内无心跳（卖家可能已长期离线）
- 缓存淘汰：listings 超过 `maxListings` 时，按时间戳删除最旧的

### 3.7 事件与回调系统

**三类事件：**

| 类型 | 注册方式 | 触发方式 | 用途 |
|------|----------|----------|------|
| 频道消息 | `TM:RegisterHandler(msgType, fn)` | 收到 `#TYPE$payload` 时自动分发 | 处理 #P/#C/#W/#X/#H/#S/#D |
| 交易事件 | `TM:RegisterTradeHandler(event, fn)` | WoW 交易事件触发 | TRADE_SHOW/ACCEPT/CANCEL/CLOSED |
| UI 回调 | `TM:RegisterUICallback(name, fn)` | `TM:RefreshUI(name)` 或 `RefreshAllUI()` | 数据变更后刷新界面 |

**初始化流程：**
1. `VARIABLES_LOADED` — 加载 SavedVariables，初始化默认配置
2. `PLAYER_ENTERING_WORLD` — 获取玩家名，两个事件都触发后开始初始化
3. 初始化各模块 → 查找/加入频道 → 频道就绪后启动同步链

### 3.8 UI 框架树

```

├── 标题栏: "龟市 TurtleMarket"
├── 关闭按钮 (右上角)

    ├── TM_PostContent — 物品预览 + 价格输入 + 备注 + 发布按钮
    ├── TM_MyListingsContent — 出售列表(8行) + 求购列表(6行) + 操作按钮
```

## 四、已知问题与改进方向

### P0 — 影响功能正确性

| # | 问题 | 位置 | 影响 |
|---|------|------|------|
| 1 | 定时器 pcall 捕获异常后，重复定时器未注销，崩溃后静默失效 | `libs.lua` 定时器 OnUpdate | 心跳/清理/同步定时器可能悄无声息停止工作 |
| 2 | 频道消息处理器已加 pcall 隔离（已修复），但高频畸形消息仍会产生大量错误日志 | `market-core.lua` handler 分发 | 性能可优化，加限流或静默计数 |

### P1 — 影响用户体验/安全

| # | 问题 | 位置 | 建议 |
|---|------|------|------|
| 3 | 信誉系统无交易验证，无频率限制，可被恶意刷分 | `market-storage.lua` UpdateReputation | 校验近期交易记录、限制每对玩家每日评价次数 |
| 4 | 交易价格不匹配仅红色警告，不阻止确认 | `market-trade.lua` | 现状合理（允许议价），可增加二次确认弹窗 |
| 5 | 求购缺少卖家主动通知 — 新商品上架时不检查匹配的求购 | `market-storage.lua` AddListing | 在 AddListing 时遍历 myWants 做匹配提醒 |
| 6 | 分片系统可被 DoS — 恶意节点发送大量不同 msgId 的分片占用内存 | `market-protocol.lua` | 已有 50 并发 msgId 上限，可加每节点限流 |

### P2 — 代码质量/UX 细节

| # | 问题 | 建议 |
|---|------|------|
| 7 | 选中状态跨 Tab 泄漏 — 切换 Tab 时旧选中行背景色残留 | 在 Tab OnHide 中清空 selectedSellId/selectedWantId |
| 8 | 银/铜输入框允许 999（MaxLetters=3），实际应限 0-99 | 发布时 clamp(0, 99) 或 OnTextChanged 校验 |
| 9 | 常量分散在各模块（协议/存储/UI 各有局部常量） | 集中到 `TM.const` 统一管理 |
| 10 | 协议消息无版本号字段，未来格式变更将导致新旧客户端不兼容 | 在消息头添加版本号如 `#P1:...` |
