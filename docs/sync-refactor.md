# 同步机制重构：思考过程与设计原则

## 背景

TurtleMarket 是 P2P 去中心化交易插件，所有通信通过 WoW 隐藏聊天频道广播。没有服务器，每个玩家客户端既是发送者也是接收者。

插件规模：最多 100 人使用，每人约 10 条商品，总计约 1000 条。

## 原始问题

旧版同步机制（#S 响应）存在四个问题：

1. **摘要算了没用** — 每个玩家上线时计算数据指纹（count:hash）发给别人，但收到的人根本不看，直接把所有数据全发一遍
2. **求购没打包** — 出售商品可以 10 条打一个包（#D），但求购是一条一条单独发的，20 条求购 = 20 条消息
3. **全量推送** — 不管对方已经有什么，每次都发全部数据。像你问朋友"有什么新消息"，他把从认识你到现在每句话都复述一遍
4. **概率响应不靠谱** — 收到同步请求后掷骰子，40%概率才回。人少的时候运气不好就收不到任何数据

## 思考过程

### 第一个方案：每人只发自己的（被否决）

想法：新玩家发 #S 后，每个在线的人只发自己发布的商品，不转发别人的缓存。

100 人 x 10 条自己的 = 1000 条，零重复，很高效。

**被否决的原因：离线玩家的数据看不到了。** 商品有 48 小时有效期，玩家下线了但商品还没过期，应该被看到。但离线玩家的数据只存在于在线玩家的缓存里，必须有人替他转发。

### 关键矛盾

需要有人转发离线玩家数据，但不能所有人都转发全量数据，否则消息爆炸（30 人在线 x 1000 条缓存 = 30000 条消息）。

### 最终方案：数据最多的人优先转发

核心原则：**谁数据最全谁先发。**

- 每个人都响应 #S 请求（去掉概率过滤）
- 每个人发自己的数据 + 缓存中别人的数据
- 但延迟时间不同：**数据多的延迟短（3秒），数据少的延迟长（20秒）**
- 数据最全的人最先发出去，后到的重复数据被去重跳过

去重已有现成机制：`if not TM_Data.listings[data.id] then` — ID 已存在就跳过。

### 数据接力问题

担忧：A 数据最全，同步给所有人后 A 下线了，新来的 B 怎么办？

答案：A 同步给 C D E 之后，C D E 手上的数据和 A 一样全。A 下线后，C D E 中数据最多的人会转发给 B。数据像接力棒一样传递，**只要任何时刻至少一个人在线，数据就不会断。**

### 重复消息问题

所有人都发数据，必然有大量重复。但不是问题：

- 接收端按 ID 去重，已有的直接跳过，开销极小
- 数据多的人延迟短、先到先写入
- 数据少的人延迟长、到的时候已经被去重了
- 不需要复杂的协调机制

## 设计原则

### 原则 1：按规模设计，不过度工程

100 人的插件不需要分布式共识、增量同步、摘要比对这些复杂机制。全量推送 + 去重就够了。

### 原则 2：数据量决定优先级

不用概率、不用选举、不用协调。只用一个简单规则：你数据越多，你越先发。自然就实现了"最全的人负责转发"。

### 原则 3：离线数据必须可见

商品有有效期（48小时），下线不等于商品失效。在线玩家的缓存就是离线玩家的代理，必须有人替他们转发。

### 原则 4：去重兜底，不怕重复

宁可多发一些重复消息，也不搞复杂的去重协调。接收端的 ID 检查（O(1) 哈希表查找）足够高效。

### 原则 5：失败要有保底

30 秒后检查：如果还是空的且知道有人在线，自动再请求一次。简单粗暴但有效。

## 防风暴优化（待实现）

### 当前问题

虽然延迟机制让数据最全的人先发，但**所有人最终都会发**。30 个在线用户全部响应 #S，大量重复 #D 消息挤占频道，挤压正常的 #P/#C 消息。

### 方案：听到 #D 就取消

在延迟机制基础上增加一个取消机制：

1. 收到 `#S` 后，照常计算延迟，设置定时器，**但将定时器 ID 存为模块变量**
2. 等待期间如果收到别人发的 `#D`，说明已有人在同步了，**取消自己的定时器**
3. 数据最全的节点延迟最短（3秒），最先发出 `#D`
4. 其他节点还在等（10-20秒），听到 `#D` 后自动取消
5. 最终只有 1-2 个节点实际发送，其余静默

### 为什么不用概率响应

概率机制会随机筛掉节点，可能把数据最全的节点筛掉，让数据少的节点回复。延迟+取消方案能**确保数据最全的节点优先**，不靠运气。

### 边界情况

两个节点数据量接近时（如 900 vs 920），延迟差极小，谁先发不确定——但无所谓，两者数据本来就差不多。

### 实现要点

修改 `market-sync.lua`：

- `#S` handler 中将 `TM.timers.delay()` 返回的定时器 ID 存为模块级变量 `syncDelayTimerId`
- `#D` handler 中检测到非自己发的 `#D` 时，调用 `TM.timers.cancel(syncDelayTimerId)` 取消发送

## 具体改动（已完成部分）

### 改了什么

**文件：`market-sync.lua`**

1. **#S 响应处理器** — 去掉 5 分钟冷却、40% 概率过滤，改为所有人响应，按数据量算延迟
2. **SendSyncData** — 发自己的 + 缓存中别人的，出售和求购都走 #D 打包
3. **#D 格式** — 新增 `L;`/`W;` 前缀区分出售和求购，兼容旧版无前缀
4. **RequestSync** — 简化为 `#S$`，去掉摘要
5. **40 秒重试** — T+40s 检查如果空的就再问一次

**文件：`libs.lua`**

常量替换：`SYNC_RESPONSE_COOLDOWN`/`SYNC_RESPONSE_CHANCE`/`SYNC_DELAY_MIN`/`SYNC_DELAY_MAX` → `SYNC_DELAY_BASE`/`SYNC_DELAY_RANGE`/`SYNC_MAX_ESTIMATE`

### 没改什么

- 实时广播（#P/#C/#W/#X）— 没问题不用动
- 上线同步链（T+3s/10s/15s/18s）— 没问题不用动
- 心跳机制（每 300 秒）— 没问题不用动
- 定期清理（每 30 分钟）— 没问题不用动
- ChatMOD 竞争 Hook — 不受影响（改动不涉及 market-core.lua 的 Hook 代码）

## ChatMOD Hook 兼容性确认

这次改动**不影响** ChatMOD 的 Hook 竞争处理，因为：

1. 三层消息过滤代码在 `market-core.lua`，本次完全没修改
2. `AddMessage` Hook 的过滤正则 `#[PCHSWXDF]%$` 仍然匹配 #D 消息 — 新增的 `L;`/`W;` 前缀在 `$` 分隔符之后，不影响匹配
3. 频道名过滤 `'urtleMarket'` 不受影响
4. `ChatFrame_OnEvent` 和 `ChatFrame_MessageEventHandler` 的双层 Hook 不受影响
5. `/leave` 命令拦截不受影响

## 延迟计算公式

```
delay = SYNC_DELAY_BASE + (1 - myCount / SYNC_MAX_ESTIMATE) * SYNC_DELAY_RANGE
```

| 本地缓存量 | 延迟 | 含义 |
|-----------|------|------|
| 1000 条（满） | 3 秒 | 数据最全，第一个发 |
| 500 条 | 11.5 秒 | 中等 |
| 100 条 | 18.3 秒 | 数据少 |
| 0 条 | 20 秒 | 没数据，最后发（其实发了也没东西） |
